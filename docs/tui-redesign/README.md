Five design directions for the Ink terminal dashboard (`packages/tui`), each rendered as a
mockup screenshot (`v*.png`) with a plain-text capture (`v*.txt`). Mockups are generated by
`gen.mjs` (writes HTML + text, and a `shots.sh` that screenshots each HTML file with headless
Chromium). All use the same fake snapshot data so directions are comparable.

Regenerate with:

```sh
node gen.mjs && bash shots.sh   # requires a chromium binary; see shots.sh
```

## Baseline critique (current TUI, `packages/tui/src/index.tsx`)

- Millisecond countdowns (`in 23.000s`, `34.000s`) repaint at 4 Hz: churn with no information,
  and false precision.
- `PID`, `SLOT`, and the truncated `SESSION` column spend ~30 columns on values an operator
  can't act on from a read-only view; meanwhile issue **titles** and **worker hosts** ‚Äî which the
  snapshot carries ‚Äî are absent, so you must know `ENG-2014` by heart.
- Bare `‚îÇ` filler lines and always-rendered empty sections burn ~8 rows.
- `runHistory` and `recentEvents` exist in `RuntimeSnapshot` but are never rendered: completed
  and failed runs simply vanish from the terminal.
- Row color is keyed to `lastEvent` type (green=turn_started, magenta=turn_completed,
  blue=other, red=none) with no legend ‚Äî it reads as decoration.
- The whole UI is one `<Text>` string re-rendered every 250 ms: Ink's reconciliation is
  defeated, and no interactivity is possible in this architecture.
- The table divider is a hard-coded 111-char string, independent of the header it underlines.

## V1 ‚Äî Flight board (`v1.png`)

One unified pipeline table: every issue in any lane (`run` / `rsv` / `retry` / `block`) is a row
with a lane tag; two-line KPI strip on top, one-line event ticker at the bottom.

**For:** everything is one scan; lanes become comparable; titles and hosts finally visible;
constant row shape keeps eye position stable; still a pure formatter (no interactivity needed).

**Against (adversarial):** the unified schema is a lie ‚Äî retry/blocked rows have five `‚Äî` cells,
so ~40 % of the grid is dead ink. Lane transitions reorder rows, defeating the "stable eyes"
claim. The activity column truncates the only human-language information. Assumes ~126 columns;
collapses in a tmux split. One ticker line loses history the moment two events happen in a poll
interval. At 10 running + 10 queued the table exceeds a screen with no paging story.

## V2 ‚Äî Mission control (`v2.png`)

k9s/lazygit-style interactive panes: KPI bar, selectable agent list, detail pane for the
selected run (host, session, workspace, sparkline, recent messages), event log, keybinding bar
(`kill`, `retry now`, `pause dispatch`, `open issue`).

**For:** detail-on-demand solves truncation forever; verbs turn the TUI from a poster into a
tool; scales via selection instead of vertical space; events keep context.

**Against (adversarial):** by far the biggest lift ‚Äî the current renderer is a string formatter;
this needs a focus model, key handling, alt-screen management, and Ink flicker/perf work at
nested-box density. Borders spend ~14 % of the area on chrome. It duplicates the web dashboard's
job over the same snapshot ‚Äî maintaining two interactive UIs is a strategy question, not a
layout question. The cramped left pane is an index, not a view: most of the time you stare at
one agent's detail while another fails unseen. Single-key `k` = kill is an accident magnet.
Useless below ~100 columns.

## V3 ‚Äî Tape (`v3.png`)

Log-first inversion: the humanized event feed is the content (newest at bottom, terminal
scrollback is history), with a sticky two-line status footer (tmux-style) carrying lane counts,
KPIs, and one compact chip per running agent.

**For:** matches what operators actually do ‚Äî watch what just happened; scrollback gives free,
greppable history and post-mortems; works at any terminal width; renders via append + tiny
footer, so Ink `<Static>` fits naturally and full-screen repaints disappear.

**Against (adversarial):** no at-a-glance table ‚Äî "how long has ENG-2027 been running?" means
squinting at footer chips that overflow past ~4 agents. Tool-call events will bury state
transitions without severity filtering, and filtering means config. Ten interleaved agent
streams are genuinely confusing. Preserving scrollback while pinning a footer is fragile ANSI
territory (resize, wrap); it's simple only if done very carefully. Chips duplicate feed content.

## V4 ‚Äî Cards (`v4.png`)

One 4-line card per running agent: title, agent/host, stage + spinner + turn/age, token
sparkline, tool count, humanized last activity. Waiting/blocked/recent compressed into a
two-line strip below.

**For:** the most readable per-agent presentation; room for the humanized message (the thing
people actually read); title + host + stage in one glance; trivially degrades to a plain list.

**Against (adversarial):** vertical cost is brutal ‚Äî 10 agents ‚âà 40+ lines before the queues.
Borders are pure chrome. A sparkline of cumulative tokens is monotonic by construction ‚Äî
decoration masquerading as data; it needs to be a *rate* to mean anything. The waiting/blocked
strip demotes failures ‚Äî arguably the highest-value information ‚Äî to a footnote. Cards also make
cross-agent comparison harder: scanning one column beats reading three prose boxes.

## V5 ‚Äî Evolution (`v5.png`)

Keep the current tree skeleton and fix the flaws: titles + hosts in the running table; PID/SLOT
/SESSION dropped; a rate-limit gauge; coarse countdowns; reserving/retry/blocked merged into one
"waiting" section with per-row reasons; a "recent" section from `runHistory`; last event as the
closing line; empty sections collapse.

**For:** cheapest to ship ‚Äî same architecture, pure formatter change; fixes every concrete
baseline complaint; no new failure modes; keeps the tree-glyph brand.

**Against (adversarial):** entrenches the fundamental limitation ‚Äî a 4 Hz full-repaint poster
with no interactivity and no scaling story past ~15 rows. Merging reserving/retry/blocked into
"waiting" blurs states the orchestrator treats very differently. Event visibility is still one
footer line. The gauge needs 256-color/truecolor support the current `s()` helper doesn't emit
(minor).

## Other directions considered (no mockup)

- **V5 + V3 hybrid (recommended):** keep the board pinned at the bottom via Ink `<Static>`
  semantics and stream humanized events into scrollback above it. Best value/effort ratio.
- **Responsive tiers:** a column-priority list so the same board renders at 80 / 110 / 140 cols
  instead of assuming one width.
- **`lorenz tail ENG-2027`:** per-agent follow as a subcommand ‚Äî the unix answer to V2's
  interactivity without building a focus model.
- **Semantic color:** color encodes lane/state, not `lastEvent` type; one legend line.
